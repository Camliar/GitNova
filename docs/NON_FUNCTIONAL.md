# Non-functional Requirements

## 隐私与安全

- 默认不离开本机；不存在 GitNova 中心服务器。
- 不记录仓库内容、diff、凭据、token 或命令敏感参数。
- 凭据委托给操作系统、System Git 或 `gh` 的既有机制。
- 网络访问必须由明确功能触发，可识别、可取消并遵循最小权限。

## 可靠性

- Host 崩溃或退出时应终止其 Core 子进程；Core 异常不得损坏仓库。
- 写操作必须可审计，失败时返回稳定错误并尽可能保持可恢复。
- SQLite 使用显式迁移；派生数据可安全重建。
- JSON-RPC 请求支持超时、取消和协议版本协商。

## 性能预算（初始目标）

- Core 冷启动 p95 目标不超过 500 ms；真实进程测试与 anti-flake ceiling 见[质量基线](QUALITY_BASELINE.md)。
- UI 输入响应目标不超过 100 ms；长任务异步执行并显示进度。
- 大型仓库操作应支持流式或分页，避免一次性加载无界数据。

## 可访问性与国际化

- 交互目标遵循 WCAG 2.2 AA；键盘可达、焦点可见、颜色非唯一信息载体。
- 文案可本地化，布局允许文本扩展；日期、时间与数字使用区域格式。
- UI 细节见[界面指南](UI_GUIDELINE.md)。

## 可维护性与兼容性

- 支持 Windows、macOS、Linux 的 Desktop 目标；具体版本矩阵在发布阶段锁定。
- 协议与持久化格式版本化；破坏性变更须迁移方案和 ADR。
- 构建应可复现，依赖可审计，关键路径有自动测试。

这些要求由[架构](ARCHITECTURE.md)、[技术栈](TECH_STACK.md)和[编码规范](CODING_STANDARD.md)共同落实。

可执行命令、网络检查和诚实的平台验证矩阵见[质量基线](QUALITY_BASELINE.md)。
